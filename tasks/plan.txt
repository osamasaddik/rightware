STEP 7: Create Admin model (src/models/Admin.ts)
  Purpose:
    Admins are seeded once at app startup. There is no admin CRUD API
    and no role system — all admins have identical full access.

  Fields:
    - email:       String, required, unique, lowercase, trim
    - password:    String, required  (stored as bcrypt hash, NEVER plaintext)
    - name:        String, required
    - timestamps:  true

  Index:
    - email (unique)

  Pre-save hook:
    - if (this.isModified('password'))
        this.password = await bcrypt.hash(this.password, 10)
    - Ensures password is always stored hashed, even if set directly

  Instance method:
    - comparePassword(plain: string): Promise<boolean>
        return bcrypt.compare(plain, this.password)
    - Used during login to verify credentials without ever decrypting

  Important:
    - NO CRUD routes for this model
    - NO role field needed (single flat access level)
    - Admins are created only via seed

STEP 8: Create Partner model (src/models/Partner.ts)
  Purpose:
    Partners are external clients that call the partner API.
    Each partner has a unique API key stored hashed in the DB.
    Created via seed script only — no CRUD API.

  Fields:
    - name:            String, required  (e.g., "Acme Corp")
    - apiKey:          String, required  (bcrypt hash of the raw key)
    - apiKeyPrefix:    String, required  (first 8 chars of raw key, plaintext)
                       Used to narrow DB lookup before bcrypt comparison
    - isActive:        Boolean, default: true
    - timestamps:      true

  Indexes:
    - apiKeyPrefix (for fast prefix-based lookup)
    - apiKey is NOT indexed (unhashable for direct query)

  Static method:
    - findByRawApiKey(rawKey: string): Promise<Partner | null>
        const prefix = rawKey.slice(0, 8)
        const candidates = await Partner.find({ apiKeyPrefix: prefix, isActive: true })
        for (const partner of candidates) {
          if (await bcrypt.compare(rawKey, partner.apiKey)) return partner
        }
        return null
    - Why prefix? bcrypt hashes cannot be queried directly.
      Storing a short plaintext prefix narrows the DB query to ~1 candidate
      before running the slow bcrypt.compare.

  Important:
    - Raw API key is shown ONCE during seeding (log it to console)
    - Only the bcrypt hash is stored in the database
    - NO CRUD routes for this model
    - Partners are created only via the seed script

PHASE 4 — AUTH MIDDLEWARE (⏱ ~30 min)
-----------------------------------------------------------

STEP 16: Create Admin JWT middleware (src/middleware/adminAuth.ts)
  Purpose:
    Protects all internal admin routes (captains, orders, assignment, reports).
    Any request without a valid JWT is rejected before reaching controllers.

  Logic:
    1. Read header: Authorization
    2. If missing or not "Bearer <token>" format → 401 "Missing or malformed token"
    3. Extract token = header.split(' ')[1]
    4. jwt.verify(token, process.env.JWT_SECRET)
    5. If TokenExpiredError  → 401 "Token expired"
    6. If JsonWebTokenError  → 401 "Invalid token"
    7. Attach decoded payload: req.admin = { id, email, name }
    8. next()

  Applied at the top of each protected router:
    router.use(adminAuth)
    → All routes in that file automatically require a valid JWT

STEP 17: Create Admin Login endpoint
  File: src/routes/auth.routes.ts + src/controllers/auth.controller.ts
  This is the ONLY public endpoint — no JWT required to call it.

  POST /auth/login
    Body: { email: string, password: string }
    Logic:
      1. Find admin: Admin.findOne({ email: body.email.toLowerCase() })
      2. If not found → 401 "Invalid credentials"
         (do NOT say "email not found" — prevents user enumeration)
      3. await admin.comparePassword(body.password)
      4. If false → 401 "Invalid credentials"
      5. Sign token:
           jwt.sign(
             { id: admin._id, email: admin.email, name: admin.name },
             process.env.JWT_SECRET,
             { expiresIn: process.env.JWT_EXPIRES_IN }
           )
      6. Return 200: { token, admin: { id, email, name } }

  Validation rules applied to this route:
    - email: isEmail(), normalizeEmail()
    - password: notEmpty()

STEP 18: Create Partner API key middleware (src/middleware/partnerAuth.ts)
  Purpose:
    Protects all /partner/v1/* routes. Only requests with a valid
    hashed API key matching an active partner in the DB are allowed.

  Logic:
    1. Read header: x-api-key
    2. If missing → 401 "API key required"
    3. const partner = await Partner.findByRawApiKey(rawKey)
    4. If null → 401 "Invalid API key"
    5. Attach: req.partner = { id: partner._id, name: partner.name }
    6. next()

  Applied at the top of the partner router:
    router.use(partnerAuth)

  Security note:
    Partner routes do NOT accept admin JWT.
    Admin routes do NOT accept partner API key.
    The two auth schemes are fully separate.

STEP 34: Implement location socket handler (src/sockets/location.socket.ts)

  AUTH STRATEGY:
    Captains  → identify by captainId (verified against DB)
    Admins    → authenticate via JWT passed in socket handshake:
                  socket.handshake.auth.token  (preferred)
                  socket.handshake.query.token (fallback)
                Both use the same JWT_SECRET as the HTTP adminAuth middleware.

  CAPTAIN EVENTS:
    Client → "captain:authenticate"
      Payload: { captainId: string }
      Server:
        1. Captain.findById(captainId)
        2. If not found or status !== 'active' → emit "auth_error", return
        3. socket.join("captain:<captainId>")
        4. emit "authenticated" { success: true, captainId }

    Client → "captain:location_update"
      Payload: { captainId: string, lat: number, lng: number }
      Server:
        1. Validate lat ∈ [-90,90], lng ∈ [-180,180] → emit "validation_error" if bad
        2. Re-fetch captain from DB, check still active
        3. Captain.findByIdAndUpdate(captainId, { currentLocation: { lat, lng, updatedAt: now } })
        4. Emit "location:updated" to room "admins"
        5. If captain has active order → also emit to room "order:<orderId>"

  ADMIN EVENTS:
    Client → "admin:join"
      Payload: {}
      Server:
        1. token = socket.handshake.auth.token
        2. jwt.verify(token, JWT_SECRET)
        3. If invalid/expired → emit "auth_error" { message }, socket.disconnect()
        4. socket.join("admins")
        5. emit "admin:joined" { success: true }

  SERVER → CLIENT EVENTS (all sockets):
    "authenticated"    { success: true, captainId }
    "auth_error"       { message: string }
    "admin:joined"     { success: true }
    "location:updated" { captainId, lat, lng, updatedAt }
    "validation_error" { message: string }

STEP 35: Enforce inactive captain check in socket handler

PHASE 12 — REPORT API (⏱ ~45 min)
-----------------------------------------------------------

STEP 38: Implement report.service.ts → getOrderVolumeDrop()
  Query params accepted:
    previousFrom, previousTo, currentFrom, currentTo,
    minPreviousOrders, minDropPercentage,
    page, limit, sortBy, sortOrder

  Validation:
    - All dates: isISO8601()
    - sortBy: isIn(['dropPercentage','dropCount','previousOrders','currentOrders'])
    - minPreviousOrders >= 1
    - minDropPercentage ∈ [0, 100]

  MongoDB aggregation pipeline:
    Stage 1: $match { captainId: { $ne: null } }
    Stage 2: $facet
      "previous": [
        { $match: { createdAt: { $gte: previousFrom, $lte: previousTo } } },
        { $group: { _id: "$captainId", previousOrders: { $sum: 1 } } }
      ]
      "current": [
        { $match: { createdAt: { $gte: currentFrom, $lte: currentTo } } },
        { $group: { _id: "$captainId", currentOrders: { $sum: 1 } } }
      ]
    Stage 3: $project to merge both facets by captainId (_id)
    Stage 4: $lookup Captain → captainName
    Stage 5: $project
      captainId, captainName, previousOrders, currentOrders,
      dropCount = { $subtract: ['$previousOrders', '$currentOrders'] }
      dropPercentage = (dropCount / previousOrders) * 100
    Stage 6: $match
      previousOrders >= minPreviousOrders
      dropPercentage >= minDropPercentage
      dropCount > 0
    Stage 7: $facet (pagination)
      "data":  [ $sort, $skip, $limit ]
      "count": [ { $count: "total" } ]
    → returns data + total in a single aggregation round-trip

STEP 49: Write ASSUMPTIONS.md
  - Admins: seeded at startup, no CRUD, no roles, full access
  - All admins are equal (flat permission model — no super-admin needed)
  - Partner API keys: hashed in DB, raw key shown once at seed time
  - Delivered/cancelled orders cannot be re-assigned (new order required)
  - Captain must be ACTIVE + ONLINE to receive assignment
  - Duplicate partner order (same partnerId + externalReference) is idempotent
  - Phone validation pattern: /^\+?[0-9]{7,15}$/
  - Socket admin auth uses same JWT as HTTP (one consistent security model)
  - Text search uses MongoDB $text index (not Elasticsearch)
  - Pagination defaults: page=1, limit=20, max limit=100

STEP 50: Write TRADEOFFS.md
  - JWT vs sessions: JWT is stateless and simpler; tradeoff is no
    server-side revocation. Acceptable for an internal admin tool.
  - Hashed partner API keys: more secure than plaintext storage;
    prefix trick keeps lookup fast. Production: use a secrets manager.
  - Admin seeded at startup vs registration flow: appropriate for
    this scope; production would need an invite/email-verification flow.
  - bcryptjs vs argon2: bcrypt is simpler and well-understood;
    argon2 is stronger but adds unnecessary complexity here.
  - MongoDB vs SQL: Mongo fits location data and flexible schemas well;
    tradeoff is more complex joins/aggregations for the report query.
  - Single static API key per partner vs OAuth2: much simpler;
    production would use per-client OAuth2 client credentials flow.
  - No location history: only latest location stored for write speed;
    history would need a separate time-series collection with a TTL index.
  - Report uses aggregation pipeline: powerful but complex; in production
    this could be offloaded to a read replica or a materialized view.
