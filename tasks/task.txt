design and implement clean APIs

handle real business rules

build search/filter/sort functionality

implement a real-time feature using sockets

correctness

Track A — Delivery Operations Backend
Scenario
You are building a small backend service for a delivery operation. Internal admins manage captains and orders. Captains can send live location updates, and partners can create orders through a simple external API.

Main Entities
Captain
Suggested fields:

id

name

phone

vehicleType

status (active, inactive)

availability (online, offline)

currentLocation:

lat

lng

updatedAt

createdAt

updatedAt

Order
Suggested fields:

id

orderNumber

customerName

customerPhone

region

fullAddress

location:

lat

lng

status (created, assigned, picked_up, delivered, cancelled)

captainId (nullable)

externalReference (nullable, for partner API)

createdAt

updatedAt

Core Requirements
1) Captain CRUD
Implement CRUD endpoints for captains.

Business rules:

inactive captains cannot be assigned to orders

inactive captains cannot send live location updates

deactivation/reactivation should be supported cleanly

2) Order CRUD
Implement CRUD endpoints for orders.

Business rules:

an order can be assigned to only one captain

delivered/cancelled orders should not be re-assigned unless you explicitly document your chosen behavior

order details endpoint must return the full location/address information

3) Assignment
Implement endpoints/services to:

assign a captain to an order

unassign a captain from an order

4) Advanced Search / Sort / Filter
Implement an orders list endpoint with:

Filtering
by status

by region

by captainId

by date range

by assignment state (assigned / unassigned)

Search
Text search on at least:

orderNumber

customerName

customerPhone

Sorting
Allow sorting by at least:

createdAt

updatedAt

status

Pagination
Include:

page / limit or cursor-based pagination

total count metadata

5) Real-Time Captain Location
Implement a socket-based feature where a captain sends location updates.

Expected behavior:

captain connects/authenticates

captain sends location payload with latitude and longitude

backend validates and stores latest location

backend emits/broadcasts the update to interested listeners (for example admin room or order room)

At minimum, document:

event names

payload format

authentication approach

how listeners receive updates

6) Report API
Implement a reporting API for admins to analyze captain workload trends.

Goal
Provide a report that shows captains who were previously handling a higher number of orders, but whose assigned order count has decreased in a later period.

Required Endpoint
GET /reports/captains/order-volume-drop

Query Parameters
Support at least:

previousFrom

previousTo

currentFrom

currentTo

minPreviousOrders
Minimum number of orders in the previous period for a captain to be included

minDropPercentage
Minimum percentage drop required to include the captain

page

limit

sortBy (dropPercentage, dropCount, previousOrders, currentOrders)

sortOrder (asc, desc)

Expected Behavior
For each captain, compare:

number of assigned orders in the previous period

number of assigned orders in the current period

Return captains whose order count dropped based on the provided filters.

Example Response Fields
Each result item should include at least:

captainId

captainName

previousOrders

currentOrders

dropCount

dropPercentage

Example Logic
If a captain had:

20 assigned orders in the previous period

8 assigned orders in the current period

Then:

dropCount = 12

dropPercentage = 60

Validation Expectations
Validate at least:

invalid phone formats if applicable

invalid lat/lng values

missing required fields

invalid status transitions if you enforce them

Optional / Bonus
Only do these if time allows:

rate limiting for partner API

location history instead of latest location only

audit log for captain activation/deactivation and order assignment

simple authorization roles (admin, captain, partner)

idempotency behavior for partner create requests
